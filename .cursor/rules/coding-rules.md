# 코딩 규칙 (공통)

VibeFund 전반에 적용하는 공통 코딩 규칙과 아키텍처 원칙입니다.

---

## 1. 기본 원칙

- **언어**: 사용자 요청이 없으면 **한국어**로 응답·주석·에러 메시지 작성.
- **DRY**: 중복 제거. 공통 로직은 유틸·훅·서비스로 분리.
- **가독성**: 명확한 네이밍, 적절한 줄 길이, 필요 시 주석.
- **타입**: TypeScript 타입·인터페이스 명시. `any` 지양.
- **품질**: 버그 없는 예시 코드 제공. 예외·엣지 케이스 고려.

---

## 2. 아키텍처

### 2.1 레이어드 구조 (권장)

- **순서**: Presentation → Application/Service → Domain → Data.
- **원칙**: 상위 레이어만 하위 참조. 같은 레이어 간 직접 의존 금지.
- **적용**: API 라우트 → 서비스(또는 유스케이스) → 리포지토리/DB 클라이언트.

### 2.2 경계 준수

- API 라우트: URL·메서드 매핑, 요청/응답 변환. 비즈니스 로직 직접 넣지 않기.
- 비즈니스 로직은 서비스·도메인에 두고, 인프라(DB, HTTP)와 분리.

### 2.3 패턴 적용

- 패턴은 **필요할 때만** 적용. 같은 패턴은 프로젝트 전반에서 네이밍·위치 일치.
- 상세: `.cursor/skills/development-patterns/SKILL.md` 참고.

---

## 3. 네이밍

- **패키지**: `@vibefund/web`, `@vibefund/shared` 등 스코프 사용.
- **내부 참조**: `"@vibefund/shared": "workspace:*"`.
- **파일**: kebab-case 또는 해당 프레임워크 관례(예: Next.js 페이지는 소문자).
- **변수·함수**: camelCase. 상수·열거형은 UPPER_SNAKE 또는 PascalCase(타입).

---

## 4. 보안·환경

- **환경 변수·비밀**: `.env`(로컬) / Vercel 환경 변수(배포)에만 저장.
- **Git**: `.env`, 비밀키, 토큰 커밋 금지.
- **API**: 인증 필요 엔드포인트는 JWT/세션 검증 필수.

---

## 5. 구현 체크리스트

구현·리팩터링 시 확인:

- [ ] 레이어 경계 준수 (API → 서비스 → 데이터).
- [ ] 비즈니스 로직이 인프라에 직접 묶이지 않았는지.
- [ ] 네이밍·파일 위치가 기존 패턴과 일치하는지.
- [ ] 테스트 가능성 (의존성 주입, 인터페이스 분리).
