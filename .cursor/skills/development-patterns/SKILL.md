# 개발 패턴 (Development Patterns)

에이전트가 기능 구현·리팩터링·아키텍처 설계 시 참고할 개발 패턴을 정리한다.  
필요한 패턴만 적용하고, 프로젝트 규모와 요구사항에 맞게 선택한다.

---

## 1. 아키텍처 패턴

### 1.1 레이어드 아키텍처 (Layered)

- **언제**: 웹/API 서비스, MVP, 팀 규모 소~중.
- **구조**: Presentation → Application/Service → Domain → Data.
- **원칙**: 상위 레이어만 하위 레이어 참조. 같은 레이어 간 직접 의존 금지.
- **적용**: API 라우트 → 서비스 → 리포지토리(또는 DB 클라이언트) 순으로 호출.

### 1.2 클린/헥사고날 (Clean / Hexagonal)

- **언제**: 도메인 복잡도가 높거나, 인프라(DB·외부 API) 교체 가능성이 있을 때.
- **핵심**: 비즈니스 로직을 “포트와 어댑터” 바깥에 두고, 인프라는 어댑터로 주입.
- **적용**: Use Case(애플리케이션 서비스)는 인터페이스(포트)만 의존하고, DB·HTTP·이메일 등은 어댑터로 구현.

### 1.3 선택 가이드

| 상황 | 권장 |
|------|------|
| MVP, 빠른 출시 | 레이어드 (3~4계층) |
| 도메인 규칙이 복잡함 | 클린/헥사고날 |
| 프론트만 분리 | API + SPA 레이어드 |

## 2. 설계 패턴 (Design Patterns)

### 2.1 생성 (Creational)

- **싱글톤**: 전역 1개 인스턴스 필요 시 (DB 연결풀, 설정). 테스트 시 목 주입 가능하도록 설계.
- **팩토리 / 추상 팩토리**: 객체 생성 로직을 한곳에 모을 때. 복잡한 의존성 조합 시 유리.
- **빌더**: 선택적 필드가 많은 DTO·옵션 객체 생성 시 가독성·안정성 확보.

### 2.2 구조 (Structural)

- **리포지토리**: 데이터 접근을 추상화. 도메인은 “저장소” 인터페이스만 의존.
- **어댑터**: 기존 라이브러리/API를 현재 인터페이스에 맞출 때.
- **의존성 주입 (DI)**: 생성자/세터로 의존성 주입. 테스트·교체 용이.

### 2.3 행위 (Behavioral)

- **전략 (Strategy)**: 알고리즘(결제 방식, 정렬 규칙 등)을 교체 가능하게.
- **옵저버 / 이벤트**: 상태 변경 시 여러 구독자에게 알림 (알림, 로깅, 캐시 무효화).
- **커맨드**: 요청을 객체로 감싸서 실행·취소·큐잉 가능하게.

### 2.4 적용 원칙

- 패턴 이름을 남발하지 말고, **문제가 있을 때만** 적용.
- 같은 패턴은 프로젝트 전반에서 일관된 네이밍·위치를 유지.

## 3. 코드/구현 패턴

### 3.1 API 계층

- **라우트**: URL·메서드 매핑만. 검증(요청 바디)·인증은 미들웨어 또는 공통 레이어에서.
- **DTO**: 요청/응답 모델을 명시. 내부 도메인 엔티티를 그대로 노출하지 않는다.
- **에러 처리**: 일관된 HTTP 상태 코드·에러 메시지 형식 (예: `{ "code", "message" }`).

### 3.2 서비스 계층

- **트랜잭션**: 하나의 유스케이스는 하나의 트랜잭션 경계. 서비스 메서드 단위로 묶는 경우가 많음.
- **도메인 로직**: 서비스는 오케스트레이션, 복잡한 규칙은 도메인 모델/도메인 서비스로.

### 3.3 데이터 계층

- **리포지토리**: `getById`, `save`, `findBy...` 등 도메인 관점의 메서드. SQL/쿼리는 리포지토리 내부에만.
- **N+1 방지**: 목록 조회 시 필요하면 조인·배치 로딩·데이터 로더 등으로 한 번에 조회.

### 3.4 프론트엔드 (선택)

- **컴포넌트**: 단일 책임, 재사용 가능한 작은 컴포넌트.
- **상태**: 전역 상태는 필요한 범위만 (폼·페이지 vs 앱 전역).
- **데이터 페칭**: 서버 상태는 캐시·재검증 전략 명시 (SWR, React Query 등).

## 4. 프로젝트 적용 체크리스트

구현/리팩터링 시 다음을 확인한다:

- [ ] 레이어 경계 준수 (API → 서비스 → 데이터).
- [ ] 비즈니스 로직이 인프라(DB, HTTP)에 직접 묶이지 않았는지.
- [ ] 새 기능에 맞는 패턴이 선택되었는지 (필요 시에만 전략/옵저버/리포지토리 등).
- [ ] 네이밍·파일 위치가 기존 패턴과 일치하는지.
- [ ] 테스트 가능성 (의존성 주입, 인터페이스 분리).
