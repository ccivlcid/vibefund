# AI 이사회 사전심사 시스템 — 현재 프로젝트 통합 검토

PRD(`AI이사회사전심사시스템PRD.md`)를 현재 VibeFund 코드베이스에 반영할 때의 **정합성**, **갭**, **권장 방향**을 정리한 문서입니다.

---

## 1. 요약

| 항목 | 결론 |
|------|------|
| **방향성** | PRD의 "투자 60% + 커뮤니티 40%"와 잘 맞고, **등록 전/직후 AI 1차 필터**로 추가하기에 적합함 |
| **점수 체계** | 기존 **Vibe Score**(체험·검증·댓글)와 **AI 투자 매력도**는 **별도 지표**로 두는 것을 권장 |
| **데이터** | AI 심사용 **입력 항목**이 현재 프로젝트 폼/DB에 거의 없음 → 필드 추가 또는 "AI 심사용 추가 정보" 단계 필요 |
| **기술** | LLM API 연동(비용·지연 고려), 비동기·큐 처리 권장 |

---

## 2. 현재 프로젝트와 PRD 비교

### 2.1 프로젝트 등록 입력 항목

PRD "Step 1 – 필수 입력 항목"과 현재 구현 대비:

| PRD 필수 항목 | 현재 상태 | 비고 |
|---------------|-----------|------|
| 프로젝트명 | ✅ `title` | |
| 한 줄 설명 | ✅ `short_description` | |
| 문제 정의 | ❌ 없음 | `description`에 포함 가능하나 별도 필드 없음 |
| 타겟 고객 | ❌ 없음 | |
| 수익 모델 | ❌ 없음 | 목표 금액·리워드만 있음 |
| 현재 개발 단계 | ✅ `status` (Prototype/Beta/Live) | |
| 팀 구성 | ❌ 없음 | |
| 예상 투자 유치 금액 | ⚠️ `goal_amount`만 있음 | “예상 유치”와 “펀딩 목표”는 다를 수 있음 |
| 경쟁 서비스 | ❌ 없음 | |

**결론**: AI 이사회 분석을 하려면 **문제 정의, 타겟 고객, 수익 모델, 팀 구성, 경쟁 서비스** 등을 새로 받거나, 기존 `description`을 활용해 LLM이 요약·추출하도록 설계해야 함.

### 2.2 점수 체계

| 구분 | 현재 구현 | PRD 제안 |
|------|-----------|-----------|
| **목적** | 커뮤니티·체험 참여도 | 투자 적합성 1차 필터 |
| **계산** | `verification*10 + comments*2 + updates*1` | (버핏×0.4 + 잡스×0.3 + Karpathy×0.3), 각 0~10 |
| **저장** | `vibe_scores` 테이블 있음, 리스트/랭킹에서 실시간 집계도 사용 | **신규** `ai_board_reports` 등 저장 필요 |

**권장**:  
- **Vibe Score**: 기존처럼 랭킹·목록 정렬·“커뮤니티 검증” 노출용 유지  
- **AI 투자 매력도**: 별도 테이블·API로 저장하고, “투자자 공개 가능(A/B/C/D)”·관리자 검수·대시보드에만 사용

### 2.3 플로우

- **현재**: 로그인 → 프로젝트 생성(제목, 한줄설명, URL, 썸네일, 카테고리, 상태, 펀딩/리워드 등) → 관리자 검수(승인/반려) → 공개  
- **PRD**: Step 1 입력 → Step 2 AI 이사회 분석 → 등급에 따른 “투자자 공개 가능” 등 상태

**통합 옵션**  
1. **등록 전**: 작성 완료 시 “AI 심사 실행” 버튼 → 결과 확인 후 제출 (입력 항목 확장 필요)  
2. **등록 직후**: 프로젝트 저장 후 백그라운드에서 AI 분석 실행 → 결과를 프로젝트 상세/관리자에 표시 (기존 폼 변경 최소)

## 3. 추가 시 필요한 작업 (체크리스트)

### 3.1 데이터

- [ ] **DB**: AI 이사회 결과 저장용 테이블 (예: `ai_board_reports`)
  - `project_id`, `overall_score`, `grade`(A/B/C/D), `strengths`, `risks`, `improvements`, `calculated_at`
  - 멤버별 점수·피드백: **확장성**을 위해 `member_scores jsonb` (예: `{ "buffett": { "score": 7.2, "feedback": "..." }, "jobs": {...}, "karpathy": {... } }`) 또는 별도 `ai_board_member_scores` 테이블 권장. (멤버 추가 시 컬럼 추가 없이 대응)
- [ ] **프로젝트 확장**: PRD 필수 항목을 쌓을 컬럼 또는 JSON 필드  
  - 예: `problem_statement`, `target_customer`, `revenue_model`, `team_info`, `expected_funding_amount`, `competitors` (또는 `project_meta jsonb`)

### 3.2 백엔드

- [ ] **LLM 연동**: OpenAI/Claude 등 API 호출, **활성 이사회 멤버**별 페르소나 프롬프트·파싱 (멤버 풀: 버핏, 잡스, Karpathy, 빌 게이츠, 머스크, 피터 린치, 피터 틸 등, 운영자 추가 가능)  
- [ ] **비동기 처리**: 분석 시간이 길어질 수 있으므로 큐(예: Vercel 배경 함수, 별도 워커) 또는 “분석 중” 상태 후 콜백/폴링  
- [ ] **API**:  
  - `POST /api/v1/projects/:id/ai-board/run` (분석 실행)  
  - `GET /api/v1/projects/:id/ai-board` (결과 조회)  
  - (선택) 관리자: `GET /api/v1/admin/ai-board/reports`

### 3.3 프론트

- [ ] **등록 플로우**: 프로젝트 정보 + **AI 이사회 멤버 선택** (유저가 포함/제외 선택), “AI 심사용 추가 정보” 단계 또는 기존 폼에 필드 추가  
- [ ] **결과 UI**: PRD 5.1 요약 대시보드 + 5.2 AI 이사회 상세 리포트(활성 멤버별 카드, 멤버 수 가변)  
- [ ] **노출 정책**: 등급 A만 투자자용 목록에 노출 등 (정책 결정 후 구현)

### 3.4 운영·비용

- [ ] **환경 변수**: LLM API 키 (Git 제외, Vercel 환경 변수)  
- [ ] **비용·한도**: 프로젝트당 **활성 멤버 수**만큼 LLM 호출, 호출량·비용 모니터링 (멤버 추가 시 비용 증가 고려)  
- [ ] **재실행**: 수정 후 “다시 심사” 버튼 등

## 4. 개선 아이디어

### 4.1 PRD 문서

- **AI 이사회**: **유명인 페르소나** 기반. 멤버 풀(버핏, 잡스, Karpathy, 빌 게이츠, 머스크, 피터 린치, 피터 틸 등)은 플랫폼/운영자가 관리. **프로젝트 등록 유저**가 심사에 **포함할 멤버를 추가(선택)** 하고 **제외할 멤버**를 지정할 수 있음. 가중치는 해당 프로젝트에 선택된 멤버 기준 합 100%.
- **등급별 액션**:  
  - B: “보완 후 공개 권장” → 보완 시 재심사 링크/버튼  
  - C: “커뮤니티 검증 필요” → 현재 Vibe Score·검증 질문과 연계 문구  
  - D: “재구성 권장” → 구체적 개선 항목 3가지 등 노출  
  이렇게 PRD에 적어두면 구현 시 일관되게 적용 가능.

### 4.2 점수·노출 전략

- **하이브리드 정렬**:  
  - 메인 랭킹: 기존 **Vibe Score** 유지  
  - “투자자용” 또는 “추천” 탭: **AI 등급(A 우선) + Vibe Score** 조합으로 정렬하면, “투자 적합성 + 검증 데이터”를 동시에 강조 가능.
- **투명성**: 창업자에게 등급과 함께 “강점 3가지 / 리스크 3가지 / 즉시 개선 항목”을 보여주면, PRD 목적(구조화된 피드백)에 잘 맞음.

### 4.3 입력 부담 완화

- AI 심사용 필드를 **전부 필수**로 하면 이탈이 늘 수 있음.  
  - **선택안**:  
    - 최소한 “한 줄 설명 + 상세 설명”만으로 1차 AI 분석 실행 → 결과와 함께 “타겟 고객·수익 모델을 채우면 더 정확한 평가를 받을 수 있어요” 유도  
    - 또는 “AI 심사 받기”를 선택 옵션으로 두고, 선택한 경우에만 추가 필드 노출

### 4.4 관리자 연동

- 현재 **관리자 검수**(승인/반려)에 **AI 등급·요약**을 한 블록으로 노출하면, “투자자 공개 가능” 여부를 판단할 때 참고하기 좋음.
- 반려 시 “AI 리스크/개선 항목” 중 일부를 반려 사유 템플릿으로 쓰는 것도 가능.

## 5. 권장 적용 순서

1. **Phase 0 (설계)**  
   - PRD 5.2 중복 정리, 등급별 액션·노출 정책 문서화  
   - “등록 전 vs 등록 직후” 플로우 결정  

2. **Phase 1 (데이터·백엔드)**  
   - `ai_board_reports` 마이그레이션  
   - 프로젝트에 AI 심사용 필드 추가(또는 `description`만으로 1차 분석)  
   - LLM 호출 서비스 + `POST/GET .../ai-board` API, 비동기 실행  

3. **Phase 2 (UI)**  
   - 프로젝트 등록 시 “AI 심사 실행” 및 결과 대시보드/카드 UI  
   - 관리자 검수 화면에 AI 등급·요약 표시  

4. **Phase 3 (정책·운영)**  
   - 등급별 노출 규칙, 재심사 플로우, 비용 모니터링
